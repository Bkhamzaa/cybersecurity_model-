# Step 3: Data Preparation

## Objective of This Step

The goal of data preparation is to transform the raw `CIC-IDS-2018` CSV files into a clean, structured, and machine-learning-ready dataset.

This step is essential because intrusion detection datasets often contain:

- identifier fields that are not useful for learning
- missing values
- infinite values generated by rate-based features
- highly imbalanced labels
- inconsistent labels that need a clear classification strategy

## Preparation Strategy

For this project, the data preparation process will include:

- merging the 5 CSV files into one analytical dataset
- checking schema consistency across files
- handling missing values
- replacing infinite values
- removing irrelevant or risky identifier columns
- defining the target-label strategy
- preparing the final feature matrix for modeling

## Schema Consistency

All five CSV files share the same schema:

- `84` columns
- same header structure
- same target column: `Label`

This means the files can be merged safely for modeling after cleaning.

## Initial Data Type Understanding

A first inspection shows that the dataset contains:

- string-based identifier/context columns such as `Flow ID`, `Src IP`, `Dst IP`, `Timestamp`, and `Label`
- integer-based columns such as ports, protocol values, and several packet/flag counters
- float-based columns such as packet statistics, flow timing features, and rate-based measurements

This is useful because most of the learning features are already numeric.

## Data Quality Checks

A sample-based quality inspection was performed before full preprocessing.

Sample inspected:

- about `42,016` rows
- `84` columns

### Duplicate Rows

In the inspected sample:

- duplicate rows found: `0`

This does not guarantee there are no duplicates in the full dataset, but it is a good initial sign.

### Missing Values

The sample showed missing values in a few columns, including:

- `Flow Bytes/s`
- `Flow IAT Mean`
- `Flow IAT Std`
- `Flow IAT Max`
- `Flow IAT Min`

This indicates that missing-value handling is required before training.

### Infinite Values

The sample also showed infinite values in some rate-based features, especially:

- `Flow Packets/s`
- `Flow Bytes/s`

This is common in network-flow datasets when division-based statistics are computed.

These values cannot be used directly in machine learning and must be replaced before modeling.

## Columns to Remove or Treat Carefully

Some columns are likely not appropriate as direct learning features because they may introduce noise, overfitting, or data leakage:

- `Flow ID`
- `Src IP`
- `Dst IP`
- `Timestamp`

These columns identify flows or endpoints rather than describing generalizable traffic behavior.

Depending on the modeling strategy, `Src Port`, `Dst Port`, and `Protocol` may be:

- kept as predictive features
- encoded if necessary
- reviewed later during feature selection

## Target Preparation Strategy

The target variable is:

- `Label`

For this PFE, the label strategy should support two possible tasks:

### 1. Binary Classification

Map all labels into:

- `BENIGN`
- `ATTACK`

This is useful for the first baseline intrusion detection model.

### 2. Multiclass Classification

Keep attack labels as separate categories to classify the specific attack type.

Possible later refinement:

- merge labels containing `- Attempted` into their parent attack families

This decision should be made carefully because some attempted classes contain very few samples.

## Planned Cleaning Operations

The preprocessing pipeline should perform the following operations:

### 1. Merge Files

Combine all 5 CSV files into one dataset for analysis and training.

### 2. Standardize Labels

Trim label spacing and ensure consistent text formatting.

### 3. Replace Infinite Values

Convert:

- `+inf`
- `-inf`

into missing values (`NaN`) before imputation or row filtering.

### 4. Handle Missing Values

Possible strategies:

- remove rows with too many missing values
- impute numeric features with median values
- impute categorical values with mode if needed

For the CIC-IDS-2018 flow features, median imputation is usually a strong baseline.

### 5. Drop Non-Useful Identifier Columns

Remove:

- `Flow ID`
- `Src IP`
- `Dst IP`
- `Timestamp`

These are not expected to generalize well in a real IDS model.

### 6. Select Feature Set

Keep the traffic and statistical columns that describe packet behavior, flow timing, and protocol activity.

### 7. Train/Test Split

Split the cleaned dataset into training and testing sets using a stratified strategy so the label distribution is preserved.

### 8. Optional Scaling

Scaling may be required for some models such as:

- `SVM`
- `Logistic Regression`
- `Neural Networks`

Scaling is less critical for tree-based methods such as:

- `Random Forest`
- `XGBoost`

## Class Imbalance Consideration

The dataset is strongly imbalanced, so data preparation must support this reality.

Possible strategies:

- use class weights
- use resampling techniques such as undersampling or oversampling
- evaluate with `precision`, `recall`, and `F1-score` instead of relying only on accuracy

For the first baseline, class weighting is a practical and simple starting point.

## Data Preparation Output

At the end of this step, the expected output should be:

- one cleaned and merged dataset
- a selected set of input features
- a target variable ready for binary or multiclass classification
- a train/test split ready for machine learning

## Notes for the Arbiter Model

This step is especially important for the future arbiter stage because:

- all base models must be trained on the same cleaned feature space
- poor preprocessing would affect every model in the ensemble
- label strategy will influence how the arbiter makes final decisions
